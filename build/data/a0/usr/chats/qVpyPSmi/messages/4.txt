# Standup Workflow

## When to Use
Activate this skill when the user triggers the phrase "daily standup".

## The Process

Follow these steps precisely to execute the daily standup:

### Step 1: Scan and Summarize Workload
1. Use the `code_execution_tool` to run the open tickets script inside the Docker container to get a JSON list of all open tickets:
```bash
docker -H unix:///var/run/docker.sock exec lamp-php-1 php /a0/usr/skills/daily-standup/scripts/get_open_tickets.php
```
2. Call a subordinate with the `project_manager` profile and pass the JSON list of open tickets.
3. Instruct the project manager to review the status of corresponding git branches (using `code_execution_tool` to run `git status` and `git branch -a`).
4. Have the project manager summarize the workload, formulate an action plan, and determine which tickets require research and which are ready for development.
5. Formulate the daily agenda summary.

### Step 2: Dispatch Research Tasks
1. For tickets returned in the JSON list with the status of `toscope`, create a new adhoc task for the Tech Lead.
2. Use the `scheduler:create_adhoc_task` tool to create a new adhoc task. Set the `name` argument appropriately for each ticket.
3. In the task prompt, explicitly instruct the Tech Lead to research the ticket, add the technical details to the `developerInstructions` field (and NOT the description), ensure that any list of acceptance criteria uses unchecked markdown checkboxes (`- [ ]`), change its status to `todo`, and assign the ticket to the developer.
4. **CRITICAL:** In the task prompt, you must explicitly add an instruction that when the task is run, the agent MUST use `call_subordinate` to delegate the execution to the specific subordinate agent the ticket is assigned to.
5. Set the `system_prompt` to reflect the role of a Tech Lead. Include a strict rule instructing them to put technical information into `developerInstructions` (not `description`) and format acceptance criteria using unchecked markdown checkboxes (`- [ ]`).
6. These tasks will remain in an `idle` state until explicitly triggered via `scheduler:run_task`.

### Step 3: Dispatch Development Tasks
1. For scoped tickets returned in the JSON list with the status of `qa`, `ongoing` or `todo`, dispatch them to the development team.
2. Use the `scheduler:create_adhoc_task` tool to create a new adhoc task for each ticket. Set the `name` argument appropriately for each ticket.
3. **CRITICAL:** In the task prompt, you must explicitly add an instruction that when the task is run, the agent MUST use `call_subordinate` to delegate the execution to the specific subordinate agent the ticket is assigned to.
4. These tasks will remain in an `idle` state until explicitly triggered via `scheduler:run_task`.

### Step 4: Enforce Strict System Prompts
When creating the adhoc development tasks via `scheduler:create_adhoc_task`, you **MUST** include the following rules exactly in the `system_prompt` argument so the developer subordinate adheres to the project rules:

> You are a developer subordinate. You must adhere to the following strict rules:
> 1. Skill Loading: You must use the skills_tool to load the `php-pro` skill before beginning any PHP work, and the `javascript-pro` skill before any JavaScript work.
> 2. Association Standard: You must associate each project ticket with a .md file and a corresponding git branch using the strict naming convention: `feature/name-####`, `bugfix/name-####`, or `task/name-####`.
> 3. Version Control Discipline: You must strictly check out the associated git branch before beginning work on a ticket, and commit all resulting code changes directly to that branch upon completion of the task.

### Step 5: Report Results
After completing the dispatch, report back to the user with a formatted response detailing:
- The daily agenda and workload summary.
- How many `adhoc` tasks were created.
- Which tickets they correspond to.
- Who (which subordinate profile, e.g., 'developer' or 'tech_lead') is handling them.
